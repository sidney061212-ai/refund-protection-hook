# Refund Protection Hook 中文说明

## 一句话介绍

Refund Protection Hook 是一个部署在 X Layer mainnet 的 Uniswap v4 `afterSwap` Hook。它让新币启动池的买家在 swap 时自愿购买 24 小时、有限额度的退款保护，降低早期买入新项目代币时的下跌风险。

## 解决的问题

新币启动阶段最难解决的不是“怎么买”，而是“买完以后风险怎么被明确、约束和兑现”。很多 launch pool 依赖项目方口头承诺、社群信任或临时补偿，但这些机制通常有几个问题：

- 买家不知道自己承担的最大下行风险是多少，只能事后被动接受价格波动。
- 项目方想降低早期买家的顾虑，但不适合承诺无限赔付或人工退款。
- 传统“保底”“补偿”“白名单福利”往往在链下执行，评判标准和资金来源不透明。
- 如果保护规则写在项目合约外部，swap 结果和保护额度容易脱节，买家实际拿到多少 token 也不一定能被准确记录。
- 对 X Layer 这样的高性能 EVM 链来说，新币启动池需要更真实的 DeFi 原语，而不是只停留在展示型 demo。

Refund Protection Hook 把这个问题拆成一个明确的链上合约关系：买家可以选择是否付费购买保护；项目方或金库只承担有限、可计算、可暂停的风险；评委和用户可以直接通过合约状态验证订单、保费、退款和风险敞口。它不是无限兜底，而是一种可验证、有限额、限时的链上退款权利。

## 项目亮点

这个设计的亮点是让买家和项目方都有明确收益，而不是单方面补贴：

- 对买家来说，它相当于一个链上“后悔权”或短期下跌保护。买家只需要支付一笔明确的保险金 premium，就可以在保护期内保留退款选择。如果买入后发现项目不符合预期、市场突然下跌、或者自己后悔参与 launch，可以通过返还买到的项目代币来拿回一部分稳定币本金。退款时支付的手续费是提前写入规则的，不需要和项目方协商。
- 对项目方来说，如果项目方对自己的项目质量和短期表现有信心，就可以主动接入这个机制。它等于向市场表达“我愿意用有限金库承担明确风险”。正常情况下，大部分买家不会退款，项目方可以持续获得保险金 premium 的一部分分成；如果有人退款，退款手续费也会产生分成。也就是说，项目方不是单纯承担赔付，而是用可控风险换取更高的买家信任和持续费用收入。
- 对金库和协议来说，premium 和 refund fee 会增强储备或形成协议收入。越健康的 launch，金库越可能积累更多 reserve；压力较大的 launch，则会按透明规则消耗 reserve，风险不会无限扩大。

所以它既是买家的风险保护工具，也是项目方的信任营销工具和持续收益机制。

## 核心机制

核心机制可以理解为“给 swap 增加一个可选的、链上记录的退款权”：

1. 项目方先为某个 stable token / project token 交易对配置保护参数，包括最小/最大订单、保护时长、premium、退款费、赔付比例和总敞口上限。
2. 金库提前存入稳定币储备，作为退款保护的资金来源。保护额度不能超过金库允许的风险敞口。
3. 买家在 Uniswap v4 pool 中用稳定币买入项目代币时，通过 `hookData` 表示自己要开启保护。
4. `UniswapV4RefundProtectionAdapter` 在 `afterSwap` 回调中读取真实的 tokenIn、tokenOut、amountIn 和 amountOut，只对符合配置的买入行为记录保护订单。
5. `RefundProtectionHook` 根据本金计算 premium 和最大赔付额，从买家处收取 premium，并要求 `RefundInsuranceVault` 锁定本订单的风险敞口。
6. 买家在保护期内可以调用 `refund(orderId)`，返还本次买到的项目代币，获得扣除退款费后的稳定币赔付。订单状态变为 `REFUNDED`，锁定敞口关闭。
7. 如果买家没有退款，保护期结束后任何人都可以调用 `finalize(orderId)`，订单状态变为 `FINALIZED`，金库释放这笔风险敞口。
8. premium 和 refund fee 会进入项目方、协议和金库的 fee split，正常 launch 可以逐步增强金库储备。

这个机制的关键点是：保护不是默认强加给所有人，而是买家自愿付费选择；项目方不是无限兜底，而是用明确参数控制最大风险。

## 实现逻辑

项目实现分成五层：

1. `RefundInsuranceVault`
   负责稳定币储备、风险敞口锁定、赔付和 fee split。它只接受授权 Hook 调用 `lockExposure`、`releaseExposure`、`payout` 和 `distributeFees`，避免任意地址动用金库。

2. `RefundProtectionHook`
   是核心状态机，负责 pair 配置、premium 计算、订单记录、退款和 finalize。每个订单都有 buyer、pairId、principalPaid、tokenAmountOut、premiumPaid、maxPayout、deadline 和 status。

3. `UniswapV4RefundProtectionAdapter`
   是真正的 Uniswap v4 `BaseHook` adapter。它启用 `afterSwap` 权限，解析 `hookData`，识别买家是否 opt-in，然后从 v4 swap 的 `BalanceDelta` 推导真实输入/输出金额，并调用 core 的 `recordProtectedSwap`。

4. `UniswapV4DemoHelper`
   是主网 demo 辅助合约，用于最小成本展示真实链上流程：添加少量 mock 流动性、执行 protected v4 swap，并保留可验证交易哈希。它不是核心产品逻辑，只是为了让评委能看到真实 X Layer mainnet 操作。

5. 部署脚本
   `deploy:xlayer` 部署 MockUSDC、MockTOKEN、Vault 和 Core；`mine:hook` 按 Uniswap v4 权限位挖出满足 `afterSwap` 的 Hook 地址；`deploy:v4-adapter` 用 CREATE2 部署 adapter 并设置 authorized recorder；`initialize:v4-pool` 初始化 v4 pool；`demo:v4-mainnet` 跑链上 liquidity、protected swap 和 refund。

链上执行顺序是：

```text
buyer swap with hookData
-> PoolManager calls adapter.afterSwap
-> adapter records protected swap in core
-> core charges premium and locks vault exposure
-> buyer can refund before deadline
-> vault pays stablecoin payout and order becomes REFUNDED
```

## 风控设计

- 单笔订单最小/最大保护金额。
- 单用户时间窗口保护上限。
- 金库总风险敞口上限。
- 动态 premium：金库利用率越高，保护费越高。
- refund fee：退款时收取费用，补偿系统风险。
- 项目方、协议、金库之间有清晰 fee split。

## 为什么适合 Uniswap v4 Hook

这个功能天然适合放在 `afterSwap`：

- Hook 可以拿到真实 swap 结果，而不是依赖前端填报。
- 保护权利在交易发生后立即记录。
- 买家是否开启保护可以通过 `hookData` 传入。
- 每个 pool 可以有自己的保护配置和风险参数。
